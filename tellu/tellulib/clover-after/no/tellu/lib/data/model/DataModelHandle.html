9    /**
10    * DataHandle implementation backed by a DataModel, implementing
11    * transaction handling and subscriptions. This is the main class
12    * for interacting with data in a model. It supports both local
13    * (relative) and absolute paths for finding nodes. A relative
14    * path starts with the name of a node directly under the context
15    * of the handle. An absolute path can find any node in the whole
16    * model. A model can have several data instances, with the first
17    * one being the unnamed and default one. An absolute path can
18    * start with a colon followed by an instance name. An absolute
19    * path starts with "/" or ":/" for the main instance of the model,
20    * or ":instName/" for another instance.
21    * <p>
22    * As the model provides a transaction system, content can only be
23    * modified in a transaction. First call a startTransaction method.
24    * Then call set/add/remove/merge/change methods, which will add
25    * the operations to the started transaction but not yet carry them
26    * out. We can then call commitTransaction or cancelTransaction.
27    * These data transactions can also be stacked - make sure to call
28    * commit/cancel as many times as start, so each transaction is closed.
29    * The handle does not give direct access to the underlying nodes.
30    * The getNode method will throw an exception. Client code can
31    * subscribe to data, to be called on any value changes, using the
32    * DataUpdateSubscriber interface. Subscriber calls are done during
33    * transaction commit, and a subscriber is called if a node referred
34    * to by the path, or any child of this, was changed by the transaction.
35    * So a subscription for a group node will give calls on change in any
36    * value in the group, as well as on addition or removal of nodes
37    * in the group. It is possible to specify new transactions in a
38    * subscription callback, even adding and removing subscribers, as
39    * transactions are queued and the next transaction executed once all
40    * subscribers have been called.
41    * <p>
42    * DataModelHandle adds some DataModel-specific features to the
43    * DataHandle interface. A DataModel can have DataProviders, which are
44    * objects providing values for data in the model. A DataProviders is
45    * responsible for keeping up-to-date values in the model as long as
46    * there are subscribers interested in this data. So it is a way to
47    * manage listeners and other resources in a way which integrates them
48    * in the model and makes sure they only run when needed. Providers are
49    * activated and deactivated through the handle, with a subscriber to
50    * be called with the provided data. And DataModelHandle has methods
51    * for adding DataProcessors to the model. A DataProcessor is an object
52    * which modifies DataNodes in the model based on the values of other
53    * nodes, creating relationships between data in the model.
54    *
55    * @author Lars Thomas Boye, Tellu AS
56    */
