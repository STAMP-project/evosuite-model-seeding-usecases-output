7    /**
8    * DataHandle is an interface for accessing a hierarchical data
9    * structure. Extending the KeyValueStore interface, it lets you
10    * access data values based on keys, and these keys can be paths.
11    * The structure is a tree with values as leaf nodes. The nodes
12    * of the tree are DataNode objects. The DataHandle interface
13    * provides a layer on top of the DataNode objects, providing some
14    * higher-level operations and allowing reading values without
15    * being concerned with the node implementation. A DataHandle
16    * object is a reference to a specific node in the structure,
17    * potentially from a set of multiple such nodes. This node is the
18    * context, and paths are relative to this context node. nextNode
19    * advances to the next in the set.
20    * <p>
21    * The path format of this package is based on XPath. A local
22    * (relative) path starts with the name of a node directly under
23    * the context, and elements of the path are separated by the
24    * slash character ("/"). Most methods also allow paths with
25    * predicates placed in square brackets following the name of an
26    * element. A predicate can be an index (starting on 1) or an
27    * expression which itself can include a path. See the DataPath
28    * class for more documentation. Depending on the implementation,
29    * absolute paths may also be supported, giving access to any
30    * part of the data structure. Most methods allow null as path
31    * argument, to refer to the context node itself. If we have a
32    * value node, the KeyValueStore get-methods can use a key of
33    * null to get the value of this node.
34    * <p>
35    * As in XML, a node may have multiple children with the same
36    * name, and so a path may refer to multiple nodes. When accessing
37    * a potential set of nodes, use getDataHandle to get a reference
38    * to the nodes of this set in the form of a new DataHandle object.
39    * If the DataHandle implementation doesn't use transactions, it
40    * is also possible to get a reference to the DataNode object the
41    * DataHandle is currently referring to. It is in any case possible
42    * to get a copy of the DataNode, using getClonedNodes.
43    * <p>
44    * The DataHandle interface includes transaction handling and
45    * subscription to changes, although implementing this is not
46    * mandatory. If supported, client code can register listeners
47    * to be notified on any data updates. Such a subscription can be
48    * made for any node in the tree, and includes any updates to
49    * child nodes. If implemented, all DataHandle methods which may
50    * modify contents MUST be called in a transaction, which means
51    * startTransaction must be called first and commitTransaction
52    * or cancelTransaction must be called to execute or cancel the
53    * transaction. Relevant methods include setting and removing
54    * values as well as adding and merging nodes.
55    *
56    * @author Lars Thomas Boye, Tellu
57    */

353    /**
354    * If the handle support transactions and data subscribers, all
355    * changes to data must be done in a data transaction, initiated
356    * with a call to startTransaction. A transaction is then open,
357    * and the set/add/remove/merge/change methods can be called,
358    * adding operations to the transaction. Finally the transaction
359    * must either be executed with commitTransaction or discarded
360    * with cancelTransaction. It is possible to start a new transaction
361    * while a transaction is open. Transactions are then stacked, and
362    * it is important to call commit/cancel the same number of times
363    * as startTransaction, so that all stacked transactions are
364    * closed and processed. If transactions are not implemented, these
365    * methods have no effect. They can safely be used in any case, to
366    * be compatible with a transaction-based system.
367    * <p>
368    * For all changed data, subscribers are notified of the update.
369    * If the code performing the transaction is itself a subscriber,
370    * and don't want to get the subscriber callback after the

372    * responsible for it), give a reference to this subscriber (owner).
