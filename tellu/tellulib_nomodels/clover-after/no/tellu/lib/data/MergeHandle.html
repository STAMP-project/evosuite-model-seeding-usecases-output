8    /**
9    * A read-only DataHandle which acts as a wrapper for multiple
10    * DataHandles. It can search for content in each contained handle
11    * in order, and its also possible to access a specific handle by
12    * naming its root node. Handles are added either to the front or
13    * back of the list of contained handles. The get-methods will start
14    * by looking in the first handle and continue through the handles
15    * until a path match is found, if called with a normal path. The
16    * path can optionally start with ':' and the name of a specific
17    * node, to select the handle who's root node matches the name (the
18    * first match, if there are multiple).
19    * <p>
20    * MergeHandle also adds support for interpreting node values as
21    * references to other nodes. If we ask for a value, and the node
22    * found by following the given path is a value node who's value
23    * is a string starting with '$', the rest of the value is assumed
24    * to be a path, and we get the result of following this path. So
25    * if we ask for a node "x", and this x has value "$:r2/y" we will
26    * actually get any nodes named "y" in the handle with the source
27    * node r2.
28    * <p>
29    * This handle is read-only, so modifying operations throws an
30    * Exception and it does not have transaction and subscription
31    * handling. Iteration (nextNode) is implemented by iterating each
32    * contained handle in parallel, so multi-node handles and iteration
33    * is supported, but not recommended as it can be confusing.
34    * The contained handles can be of any type based on DataHandleImpl.
35    *
36    * @author Lars Thomas Boye, Tellu
37    */
