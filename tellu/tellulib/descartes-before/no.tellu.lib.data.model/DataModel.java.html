/**
28 
 * A DataModel manages a set of data and (optionally) a corresponding
29 
 * set of meta-data. While DataNodes can be used stand-alone, and
30 
 * also combined with MetaNodes without a DataModel, this is the class
31 
 * which ties it together and integrates it with other mechanisms.
32 
 * Most importantly it adds a transaction and subscription system for
33 
 * data value changes, to keep track of changes and notify subscribers,
34 
 * implementing these aspects of the DataHandle interface. Most data
35 
 * access is handled through the DataModelHandle implementation of
36 
 * DataHandle, so client code will usually not have direct access to
37 
 * the DataModel.
38 
 * <p>
39 
 * The model keeps one or several data node trees. The first one is
40 
 * the main and default one, and is unnamed. One data structure is
41 
 * sufficient for most uses, but the model supports having additional
42 
 * data instances, useful for cases where we need to process data from
43 
 * multiple sources/documents together while keeping them distinct for
44 
 * reading/writing from/to files etc. A colon is used to specify a
45 
 * named instance. <code>:x/y/z</code> refers to a node z in a node y
46 
 * under the root of an instance named x. Meta data can be defined for
47 
 * any instance. Meta data is optional, and we can have data without
48 
 * meta data, but if we want meta data for a node then any parent node
49 
 * also must have meta data (/a/b can't have meta data if /a doesn't
50 
 * have meta data).
51 
 * <p>
52 
 * The DataModel interface is mainly for initialization and management
53 
 * of the model, with the DataHandle interface handling data access.
54 
 * DataModel supports parsing/serialization, such as reading/writing
55 
 * XML or JSON, through DataSerializer objects. It provides the
56 
 * framework for reading/writing from streams, especially files, to
57 
 * be configured or extended in sub-classes. It can for instance be
58 
 * connected to a file, reading initial content from a file and
59 
 * persisting changes to the file every time a transaction completes
60 
 * with changes.
61 
 * <p>
62 
 * Once the model has been initialized, any operation which may change
63 
 * data values must run as a transaction. The DataHandle has the
64 
 * transaction interface for data updates. Some DataModel operations,
65 
 * such as adding meta data or resetting all values to their meta
66 
 * defaults, are complete transactions in themselves. The model has
67 
 * a mechanism for client code to subscribe to changes in data, with
68 
 * DataUpdateSubscribers registered through the DataHandles. A related
69 
 * mechanism is that of DataProvider, which is an object which
70 
 * provides values for some data in the model.
71 
 * <p>
72 
 * Another key concept is DataProcessor. This is an object which
73 
 * updates the value(s) of nodes based on changes in other nodes,
74 
 * typically representing a relationship between data, such as a
75 
 * mathematical function. The DataModel can create DataProcessors by
76 
 * parsing expressions with data paths and operators, so that it can
77 
 * implement numerical and boolean expressions from strings. With
78 
 * this expression parsing, DataModel also support defining conditions
79 
 * for rules. Adding Trigger and Action objects, DataModel can be
80 
 * used as a basic rule engine for Trigger-Condition-Action rules.
81 
 * <p>
82 
 * A typical way to use DataModel is to first add MetaData to define
83 
 * the model schema, along with any DataProcessors and DataProviders
84 
 * needed. Then we can initialize the model with either the contents
85 
 * of a file or with the default values specified by the meta. Then
86 
 * we get a DataHandle to read, write and subscribe to data.
87 
 *
